1. 목적

본 계획은 pharma-ntp-standard 시스템의 컴퓨터 시스템 검증(CSV) 접근 방식, 범위, 책임, 산출물, 합격 기준을 정의한다.

2. 시스템 개요

제약(GxP) 환경에서 NTP 시간 동기화 표준을 수립·모니터링·검증하는 웹 애플리케이션. GxP 전자 기록의 타임스탬프 신뢰성을 보장하는 것이 목적이다.

3. GAMP 5 분류 및 시스템 영향도 평가

3.1 GAMP 5 소프트웨어 카테고리

본 시스템은 GAMP 카테고리 5(맞춤 개발 애플리케이션) 로 분류한다. 표준 NTP 프로토콜(시간 소스)은 인프라(카테고리 1)에 의존하나, 모니터링·검증·표준 관리 로직은 맞춤 개발이다.

3.2 시스템 영향도 평가 (System Impact Assessment)

본 시스템은 직접 영향 시스템(Direct Impact System) 으로 분류한다. 최신 분류 체계는 '간접 영향(Indirect Impact)'을 별도 등급으로 두지 않으며, GxP 데이터 무결성에 영향을 주는 시스템은 직접 영향으로 본다.

  • 근거: 본 시스템이 생성하는 한계 초과 로그(FS-023)·정합성 보고서(VSR-001)는 GxP 전자기록 타임스탬프의 신뢰성을 입증·판단하는 근거로 사용된다. 경고 판정(RISK-001)이 오작동하면 비동기 장비가 검출되지 않아, 해당 장비가 남긴 GxP 기록의 시각 무결성(ALCOA+의 Contemporaneous)이 훼손될 수 있다.
  • 범위 경계(중요): 본 시스템은 장비 시계를 설정·보정하지 않는다 — 감시·읽기 전용(NTP mode 3 클라이언트 한정, RISK-011). 실제 시각 동기화(보정)는 인프라(NTP 서버·OS, GAMP 카테고리 1)의 책임으로 본 검증 범위 밖이다. 따라서 직접 영향의 대상은 데이터 무결성(감시·보고 기록) 이며, 시각 제어 기능은 시스템에 포함되지 않는다.
  • 결과: 직접 영향 분류에 따라 본 시스템은 전체 V 모델 검증(IQ/OQ/PQ) 대상이다. 위험 기반으로 GxP 영향도가 높은 기능(오프셋 경고·표준 무결성, RISK-001/002)에 검증 노력을 집중한다(§5 참조).

1. 개요

본 문서는 pharma-ntp-standard 시스템에 대한 사용자(품질·운영) 관점의 요구사항을 정의한다. 각 요구사항은 검증 가능해야 하며, PQ에서 입증된다.

1.1 기준 표준 (Reference Standard)

본 시스템의 신뢰 시간 기준은 한국표준과학연구원(KRISS)이 유지하는 협정세계시 실현 UTC(KRISS) 이며, KRISS 공식 시각 동기화 프로그램 UTCk(v4.1.0.2, time.kriss.re.kr 계열 NTP 소스)가 제공하는 시각을 표준 기준점으로 삼는다. 본 프로젝트는 공식 인증 제품이 아니며, UTCk를 참조 기준으로 하여 그에 준하는 모니터링·검증 체계를 제공한다.

2. 요구사항 표기

  • GxP 영향도: H(높음)/M(중간)/L(낮음) — 위험 평가와 연계

보안 측면 High 등급 RISK-007(인터넷↔폐쇄망 피벗)은 자세 A 배포의 핵심 잔여 위험으로, egress 화이트리스트·전용 호스트·내부 바인딩으로 완화하며 OQ-050에서 현장 검증한다. 이 시스템은 장비에 쓰기를 하지 않는 읽기 전용 관찰자이므로 장비 데이터·검증상태에는 영향을 주지 않는다(RISK-011 Low).

2.1 핵심 알고리즘 (NTP/SNTP 오프셋)

UTCk를 포함한 모든 NTP/SNTP 클라이언트는 4개 타임스탬프로 오프셋·지연을 계산한다(RFC 5905 / SNTP RFC 4330).

T1 = 클라이언트 송신 시각      T2 = 서버 수신 시각
T3 = 서버 응답 송신 시각       T4 = 클라이언트 수신 시각

오프셋 θ = ((T2 − T1) + (T3 − T4)) / 2
왕복지연 δ = (T4 − T1) − (T3 − T2)

UTCk의 동기화는 이 θ만큼 로컬 OS 시계를 보정하는 것이고, 측정/비교는 θ를 표시하는 것이다.

  • 표준 변경 이력(OQ-002)은 구현 완료(일탈 #2 Closed) — 표준 값 변경이 버전별 스냅샷·사유로 추적된다(RISK-002 완화). 전 엔터티 전체 감사 추적(FS-040/OQ-040)은 범위 제외(일탈 #3) — 핵심 이벤트는 한계 초과 로그(FS-023)로 기록. 인증 부재로 변경 이력의 '누가'는 system이며, 전자서명 기반 완전 통제는 후속이다.
  • 측정은 네트워크 도달성에 의존하며, 폴링 중단은 STALE 상태로 감지된다(RISK-003).

1. 목적

제약(GxP) 환경에서 NTP 시간 동기화는 IT 설정이 아니라 데이터 무결성(Data Integrity)의 법적 요건이다. 전자기록의 "C(Contemporaneous, 동시성)"와 감사추적의 신뢰성이 동기화된 보안 시계에 직접 의존하기 때문이다. 본 문서는 최신 규제 동향과 산업 모범사례를 규명하고, 이를 본 시스템의 요구사항(URS)·위험(RA)·검증(IQ/OQ/PQ)과 정합 매핑하여, 설계 결정의 외부 정당화 근거를 제공한다.

장비 NTP 설정 확인·등록 매뉴얼 (운영 가이드)

항목 내용
문서 ID MAN-001
버전 1.0
대상 QC 운영자
목적 감시 대상 장비가 NTP 질의에 응답하도록 설정·확인하고, 본 시스템에 등록·점검하는 실무 절차
관련 기능 명세 FS-010/020 · 위험 RISK-011

이 문서는 검증 산출물(V 모델)이 아니라 운영 절차서입니다. 코드/설정 환경에 맞춰 현장에서 보완해 사용하세요.

1. 먼저, 동작 원리 한 장 (왜 NTP "응답"이 필요한가)

이 시스템(감시 PC)은 각 장비에게 "지금 네 시계로 몇 시냐?" 를 NTP로 물어봅니다. 그 답을 KRISS 기준 시각과 비교해 장비 시계가 얼마나 벗어났는지(오프셋)를 측정합니다.

[감시 PC] ──"몇 시?"(NTP 질의, UDP 123)──▶ [장비]
          ◀──"○○:○○:○○"(NTP 응답)──────
   │
   └─ 같은 방식으로 KRISS에도 질의 → 두 답을 비교해 (장비 vs KRISS) 오차 산출

여기서 핵심 두 가지:

  1. 장비가 NTP 질의에 "응답"해야 합니다 = 장비에서 NTP 서버(시간 응답) 기능이 켜져 있어야 합니다. 응답하지 않으면 본 시스템에 등록 자체가 거부됩니다(아래 §6 트러블슈팅).
  2. 장비 자신은 신뢰 시간원에 동기화되어 있어야 합니다(사내 NTP/도메인 컨트롤러/KRISS 등). 본 시스템은 장비 시각을 바꾸지 않습니다(읽기 전용 감시 — RISK-011). 틀어진 시각을 발견해줄 뿐, 맞추는 것은 장비 쪽 설정의 몫입니다.

2. 사전 점검 체크리스트

장비 1대를 등록하기 전에 아래를 순서대로 확인하세요.

# 확인 항목 방법 통과 기준
1 장비 IP/호스트명 장비 네트워크 설정 고정 IP 권장
2 네트워크 도달 감시 PC에서 ping <장비IP> 응답 수신
3 NTP 응답 감시 PC에서 w32tm /stripchart /computer:<장비IP> /dataonly /samples:3 시각 오프셋이 출력됨(응답함)
4 방화벽 감시 PC → 장비 UDP 123 허용 차단 없음

3번이 실패(타임아웃)하면 장비의 NTP 서버 기능이 꺼져 있거나 방화벽에 막힌 것입니다 → §3에서 켭니다.

3. 장비 유형별 — NTP 응답 설정·확인

3.1 Windows PC / 워크스테이션 (HPLC·GC 제어 PC 등)

대부분의 GxP 분석기 제어 PC가 Windows입니다.

(A) 현재 상태 확인 — 장비에서 관리자 명령 프롬프트:

w32tm /query /status          :: 동기화 상태·마지막 동기 시각
w32tm /query /configuration   :: NTP 서버(NtpServer) 활성 여부

(B) 장비를 신뢰 시간원에 동기화 (장비 시각 자체가 맞아야 함):

w32tm /config /manualpeerlist:"<사내NTP 또는 time.kriss.re.kr>" /syncfromflags:manual /reliable:yes /update
w32tm /resync

(C) NTP 서버 응답(질의에 답하기) 활성화 — 레지스트리:

HKLM\SYSTEM\CurrentControlSet\Services\W32Time\TimeProviders\NtpServer
    Enabled = 1   (DWORD)
HKLM\SYSTEM\CurrentControlSet\Services\W32Time\Config
    AnnounceFlags = 5   (DWORD)

적용 후 서비스 재시작:

net stop w32time && net start w32time

도메인 가입 PC는 시간 정책(GPO)이 우선될 수 있습니다. AD 도메인 컨트롤러가 시간원이면 감시 대상은 DC 또는 각 PC로 잡습니다.

3.2 Linux 시스템 (chrony / ntpd)

상태 확인:

timedatectl                 # System clock synchronized: yes 인지
chronyc tracking            # 기준 소스·오프셋 (chrony)
chronyc sources -v

다른 호스트(감시 PC)의 질의에 응답 허용 — /etc/chrony/chrony.conf:

allow 10.0.0.0/24           # 감시 PC가 속한 대역

적용:

sudo systemctl restart chronyd

ntpd 사용 시: /etc/ntp.conf의 restrict 규칙으로 감시 PC 대역의 질의를 허용하고, ntpq -p로 상위 동기 상태를 확인합니다.

3.3 계측기·임베디드·네트워크 장비

  • 벤더 설정 화면(웹 UI/패널)에서 NTP 클라이언트(동기) 와 NTP 서버(응답) 항목을 확인합니다.
  • 일부 장비는 NTP 서버 응답을 지원하지 않습니다 — 이 경우 본 시스템으로 직접 감시할 수 없습니다. 대안: 그 장비가 동기화하는 상위 NTP 서버를 감시 대상으로 잡거나, 별도 절차로 시각을 주기 점검합니다.
  • 설정 방법은 벤더 매뉴얼/현장 엔지니어에게 문의하세요. 본 시스템은 읽기 전용이라 장비 설정을 강제하지 않습니다.

4. 본 시스템에 장비 등록 (FS-010)

  1. 대시보드 화면 → 장비 등록
  2. 장비 이름(예: HPLC-WS-01), IP 또는 호스트, 동기화 표준 선택 → 추가
  3. 등록 시 시스템이 자동으로 NTP 응답을 확인합니다 — 응답하는 장비만 등록되고, 무응답이면 거부(422)됩니다. (죽은 장비가 등록부에 들어가 데이터가 오염되는 것을 막는 안전장치)

등록이 거부되면 §2 체크리스트와 §3 설정으로 돌아가세요. 대량 시드 등 예외 상황에서는 검증 우회 옵션이 있으나 운영에서는 권장하지 않습니다.

5. 대시보드 상태 읽는 법

상태 배지 의미 조치
정상(OK) 오프셋이 허용 한계 이내 없음
한계 초과(BREACH) 장비 시각이 허용 한계를 벗어남 §6 — 장비의 상위 시간 동기 점검
갱신 지연(STALE) 최근 측정이 없음(폴링 노후) 스케줄러/네트워크 점검
응답 없음(UNREACHABLE) 장비가 NTP 질의에 무응답 장비 전원·네트워크·NTP 서버 점검
미측정(UNKNOWN) 아직 1회도 측정 전 다음 폴링 주기까지 대기

표의 오프셋은 KRISS 기준 편차(초), 허용 한계는 표준에 설정한 임계값입니다.

6. 자주 겪는 문제 (트러블슈팅)

증상 원인 해결
등록 시 "NTP 응답을 하지 않아 등록할 수 없습니다"(422) 장비 NTP 서버 미활성 / 방화벽 UDP 123 차단 §3에서 NTP 서버 응답 활성화, 방화벽 허용
상태가 응답 없음(UNREACHABLE) 장비 꺼짐·네트워크 단절·IP 변경 ping·전원 확인, IP 고정
상태가 한계 초과(BREACH) 장비 시각 자체가 어긋남(상위 동기 실패) 장비의 NTP 동기 소스 점검(§3 B), w32tm /resync
오프셋이 비현실적으로 큼 스푸핑/이상치 → 시스템이 미신뢰 처리 네트워크 경로·시간원 점검(이상 시 synced=false 표시)
상태가 계속 갱신 지연(STALE) 백그라운드 폴링 중단/주기 과다 감시 PC 앱 가동·폴링 주기 확인

7. 정기 점검 권장 (운영)

  • 폴링 주기·허용 한계: 대시보드 동기화 설정에서 조정(변경 시 사유와 함께 이력 기록됨 — 표준 변경 이력).
  • 한계 초과 로그: 허용 한계 초과 로그 탭에서 주기적으로 검토(최근 1주일 / 지난 이력 기간 조회).
  • 신규/이전 장비: IP 변경 시 재등록, 폐기 장비는 등록 해제.
