# 기능 명세 (Functional Specification, FS)

| 항목 | 내용 |
|------|------|
| 문서 ID | FS-001 |
| 버전 | 0.1 (Draft) |
| V 모델 대응 | 운영 적격성(OQ) |

## 1. 개요

본 문서는 [URS](01-user-requirements.md)를 충족하기 위해 시스템이 제공하는 기능을 명세한다. 각 FS는 1개 이상의 URS에서 전개되며 [OQ](08-oq-protocol.md)에서 검증된다.

## 2. 기능 목록

### 2.1 표준 관리

| ID | 기능 | 출처 URS |
|----|------|---------|
| FS-001 | 시간 표준(신뢰 소스 + 파라미터) CRUD API와 화면을 제공한다. | URS-001, URS-002 |
| FS-002 | 표준 저장 시 버전을 증가시키고 변경 이력 레코드를 생성한다. | URS-003 |

### 2.2 장비 관리

| ID | 기능 | 출처 URS |
|----|------|---------|
| FS-010 | 장비 등록/조회/수정 API와 목록 화면을 제공한다. | URS-010 |
| FS-011 | 장비에 `gxp_critical`(boolean) 분류 필드를 둔다. | URS-011 |
| FS-012 | 장비에 표준(정책) ID를 연결한다. | URS-012 |

### 2.3 모니터링

| ID | 기능 | 출처 URS |
|----|------|---------|
| FS-020 | 폴링 워커가 각 장비의 NTP 오프셋을 주기적으로 측정·저장한다. | URS-020 |
| FS-021 | 대시보드 API가 장비별 최신 오프셋·드리프트·stratum·last_sync를 반환한다. | URS-021 |
| FS-022 | 오프셋 절댓값이 표준의 허용 한계를 초과하면 `alert` 레코드를 생성하고 상태를 `BREACH`로 표시한다. | URS-022 |
| FS-023 | 경고는 생성/해제 타임스탬프를 보존하며 이력 조회 API를 제공한다. | URS-023 |

### 2.4 검증 산출물

| ID | 기능 | 출처 URS |
|----|------|---------|
| FS-030 | IQ/OQ/PQ 산출물 CRUD API와 화면을 제공한다. | URS-030 |
| FS-031 | 산출물 상태 전이(Draft→Reviewed→Approved→Effective)를 강제한다. | URS-031 |
| FS-032 | 산출물에 요구사항 ID 태그를 연결할 수 있다. | URS-032 |

### 2.5 공통 / 데이터 무결성

| ID | 기능 | 출처 URS |
|----|------|---------|
| FS-040 | 중요 엔터티(표준/장비/산출물) 변경 시 audit 레코드를 append-only로 기록한다. | URS-040 |
| FS-041 | 시스템 시각은 구성된 NTP 소스 기준으로 표시한다. `GET /api/time`이 server UTC에 KRISS 오프셋을 보정한 **기준 시각**(`reference_utc`)·stratum·동기 여부를 반환하고, 대시보드는 이를 앵커로 KST(UTC+9) 대형 시계를 표시한다. NTP 도달 실패 시 `synced=false`로 서버 시각을 폴백 표시(미검증 명시). | URS-041 |

### 2.6 보안 / 네트워크

| ID | 기능 | 출처 URS |
|----|------|---------|
| FS-050 | 백엔드/프론트는 기본적으로 루프백(localhost) 또는 내부 인터페이스에만 바인딩하며, 노출 시 인증을 요구한다. 인터넷 아웃바운드는 구성된 신뢰 소스 호스트(KRISS)·NTP(UDP 123)로만 제한하는 배포 전제를 둔다. | URS-050, URS-052 |
| FS-051 | NTP 측정은 **클라이언트 모드(mode 3) 읽기**로만 수행한다. 시각 설정·제어 모드(mode 6/7) 기능을 제공하지 않아 장비 시각·설정을 변경할 수 없다. | URS-051 |
| FS-052 | 신뢰 시간 소스로의 통신은 구성된 호스트로만 향하며, **다중 샘플 중앙값 + 합리적 범위(sanity bound) 검증**으로 스푸핑·이상치를 완화한다. 이상 시 `synced=false`로 표시한다. | URS-050 |

## 3. 비기능 요구

- 모니터링 폴링 주기는 표준당 구성 가능(기본 60초).
- 대시보드 API 응답 1초 이내(장비 500대 기준) — [PQ](09-pq-protocol.md)에서 검증.
